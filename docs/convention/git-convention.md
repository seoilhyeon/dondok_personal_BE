# Git 협업 컨벤션

## 이슈 규칙
- 작업 시작 전 반드시 이슈를 먼저 생성한다
- 이슈 생성 → 브랜치 생성 → 작업 → PR → 머지 순서를 따른다
- 이슈 제목 형식: `[feat] 크루 생성 API 구현`
- PR 제목은 이슈 제목과 동일하게 작성한다
- 이슈 본문은 `.github` 템플릿 양식을 따른다

## 브랜치 전략
- `main` : 운영 배포 (직접 push 금지)
- `feat/{이슈번호}-{작업명}` : 기능 개발 (예: feat/12-auth-jwt)
- `fix/{이슈번호}-{작업명}` : 버그 픽스
- 머지 완료된 브랜치는 자동으로 즉시 삭제

## 커밋 컨벤션 

| 타입 | 설명 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 리팩토링 (기능 변경 없음) |
| `style` | 코드 스타일 변경 (포매팅, 세미콜론 등) |
| `docs` | 문서 수정 |
| `test` | 테스트 추가/수정 |
| `chore` | 설정, 빌드, 파일명·폴더명 변경, 파일 삭제 등 기타 |
| `ci` | CI/CD 관련 설정 변경 |

커밋 메시지에는 반드시 이슈 번호를 포함한다.
예: `feat: 로그인 기능 구현 (#10)`

## PR 규칙
- 머지 대상: main 브랜치
- 최소 1명 이상 코드 리뷰 승인 후 머지
- PR 제목 형식: [feat] 크루 생성 API 구현
- PR 본문은 `.github` 템플릿 양식을 따른다

## 코드 리뷰 기준
- 24시간 이내 리뷰 완료
- `Approve` : 이상 없음, 머지 가능
- `Request Changes` : 수정 필요, 재요청 후 머지
- `Comment` : 단순 의견, 반영 여부는 작성자 판단

## CodeRabbit
- PR 생성 시 CodeRabbit이 자동으로 AI 리뷰를 수행
- CodeRabbit 리뷰는 참고용이며 사람 리뷰를 대체하지 않음
- `@coderabbitai` 멘션으로 특정 파일/라인 재검토 요청 가능
- 명백한 오탐(false positive)은 코멘트로 dismiss 처리
